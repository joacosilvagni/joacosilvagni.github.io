API Testing — Reconocimiento, Endpoints Ocultos y Mass Assignment

Metodología completa para auditar APIs REST: descubrimiento de documentación, fuzzing de endpoints y métodos HTTP, cambio de Content-Type y explotación de Mass Assignment. Incluye resolución de laboratorios de PortSwigger Web Security Academy.

API Testing Mass Assignment Burp Suite PortSwigger

Identificación de API Endpoints

Lo primero es identificar los API endpoints. Por ejemplo, si hay alguna petición que se vea así:

GET /api/books HTTP/1.1
Host: example.com

Esto es una interacción con la API. Si otra petición se ve así:

/api/books/mistery

Ya se ve la superficie de interacción con la API.

Se interpreta de la siguiente forma:

  • Host: example.com → El servidor al que se está conectando es example.com
  • GET /api/books → La ruta (path) que se está solicitando en ese servidor es /api/books

API Data Input

  • Una vez identificados los endpoints, hay que verificar cómo interactuar con ellos y qué datos procesan.
  • Qué tipos de peticiones acepta la API.

Ejemplo:

GET /api/books       → para listar libros
POST /api/books      → para crear un libro nuevo
PUT /api/books/123   → para actualizar todo el libro
PATCH /api/books/123 → para actualizar solo un campo
DELETE /api/books/123 → para borrar

Authentication — ¿Cómo se autentica esta API?

  • ¿Necesita un token JWT en el header Authorization: Bearer xxx?
  • ¿Usa API Key (X-API-Key: abc123)?
  • ¿Usa cookies de sesión?
  • ¿Permite acceso sin autenticación (público)?

API Documentation

Las APIs usualmente están documentadas para que los mismos desarrolladores entiendan su uso. Siempre empezar la API Recon buscando esta documentación.

Formas de descubrir la documentación de la API

  1. Usar Burp Scanner (el crawler automático) — Burp puede crawlear toda la aplicación automáticamente e intentar encontrar páginas ocultas, incluyendo la documentación.
  2. Navegar manualmente con el Browser de Burp — Abrís el navegador integrado de Burp y vas navegando por la aplicación normalmente. Mientras navegás, Burp captura todas las peticiones y a veces descubre endpoints de documentación.
  3. Buscar rutas (paths) comunes de documentación — Las rutas más típicas donde suele encontrarse la documentación:
    • /api
    • /swagger/index.html → abre la interfaz visual de Swagger (muy común)
    • /openapi.json → archivo en formato JSON que describe toda la API
    • /swagger.json
    • /api-docs
    • /v2/api-docs
    • /docs
    • /api/swagger

Lab: Exploiting an API endpoint using documentation

Objetivo: Eliminar al usuario "carlos" usando la documentación de la API.

Lo primero, una vez configurado Burp y el target, es iniciar sesión con la cuenta que da PortSwigger y probar funcionalidades que interactuarían con la API.

Al hacer "Update Email", aparece la siguiente petición:

PATCH /api/user/wiener HTTP/2
Host: [target].web-security-academy.net
Content-Type: text/plain;charset=UTF-8

{"email":"admin@admin.com"}

El concepto principal de este laboratorio es ir acortando la URL:

/api/user/wiener → /api/user → /api/

Y cambiar el tipo de petición de PATCH a GET para leer la documentación de la API.

Diferencias entre métodos HTTP:

Método Qué hace Uso típico
GET Obtener información Leer datos
POST Crear un nuevo recurso Crear usuario, subir archivo
PUT Reemplazar todo el recurso Actualizar un objeto completo
PATCH Actualizar solo una parte Cambiar solo el email, solo el estado
DELETE Eliminar un recurso Borrar usuario

La documentación obtenida de la API es:

Verb Endpoint Parameters Response
GET /user/[username: String] { } 200 OK, User
DELETE /user/[username: String] { } 200 OK, Result
PATCH /user/[username: String] {"email": String} 200 OK, User

Con esta información, se envía una petición DELETE a /api/user/carlos sin parámetros. Laboratorio completado.

API Endpoints — Enumeración Avanzada

1. Identificar endpoints (cómo encontrarlos)

Aunque tengas documentación de la API, siempre es bueno buscar por cuenta propia porque la documentación suele estar desactualizada o incompleta.

Cómo hacerlo en la práctica:

  • Usá Burp Scanner para crawlear toda la aplicación automáticamente.
  • Mientras navegás normalmente con Burp Browser, fijate en las URLs con patrones como:
    • /api/
    • /v1/, /v2/, /api/v3/
    • /rest/, /graphql
  • Revisá los archivos JavaScript (.js) — muchas veces los desarrolladores dejan ahí las llamadas a la API, especialmente en aplicaciones modernas con React, Angular o Vue.

Herramientas recomendadas:

  • JS Link Finder (BApp de Burp) → extrae endpoints de los archivos JS
  • Pestaña Proxy > HTTP history o Target > Site map

Consejo práctico: Después de crawlear, ir a Target → Site map, filtrar por "api" y revisar todo. También filtrar por extensión .js.

2. Interactuar con los endpoints

Una vez encontrado un endpoint (ej: /api/tasks), se juega con él usando:

  • Burp Repeater → para probar manualmente y ver respuestas
  • Burp Intruder → para automatizar pruebas (fuzzing)

Qué probar:

  • Cambiar el método HTTP (GET, POST, PUT, DELETE, PATCH, OPTIONS...)
  • Cambiar el Content-Type
  • Modificar parámetros, IDs, cuerpos de request

3. Métodos HTTP soportados

Cada endpoint puede aceptar diferentes métodos HTTP. No todos los métodos están documentados.

Ejemplo:

Método Endpoint Qué hace
GET /api/tasks Obtener lista de tareas
POST /api/tasks Crear una tarea nueva
DELETE /api/tasks/1 Borrar tarea con ID 1
PUT / PATCH /api/tasks/1 Actualizar tarea

Cómo probarlo: Enviar el request al Intruder y cargar la lista de HTTP verbs predefinida de Burp (Built-in wordlist → HTTP verbs).

Siempre probá primero en objetos de baja prioridad (por ejemplo, tareas que vos mismo creaste), nunca en datos críticos.

4. Content-Type — tipos de contenido soportados

Este es uno de los puntos más importantes en testing de APIs.

Muchas APIs esperan JSON (application/json), pero si se envía otro formato (XML, form-data, etc.) pueden comportarse de forma diferente.

Por qué es útil cambiar el Content-Type:

  • Puede revelar información sensible en los errores
  • Puede burlar validaciones o filtros (ej: WAF que solo filtra JSON)
  • Puede cambiar la lógica interna (ej: segura en JSON pero vulnerable a XXE en XML)

Cómo hacerlo:

Cambiar el header:

Content-Type: application/json

por:

Content-Type: application/xml

Y cambiar el body al formato correspondiente.

Herramienta recomendada: Content type converter (BApp de Burp) — convierte automáticamente JSON ↔ XML con un clic.

Content-Types comunes a probar:

  • application/json
  • application/xml
  • text/xml
  • application/x-www-form-urlencoded
  • multipart/form-data

Lab: Finding and exploiting an unused API endpoint

Objetivo: Comprar la "Lightweight l33t Leather Jacket" (precio ~$1337) con $0 en la cuenta.

Pasos:

  1. Login con wiener:peter
  2. Ir al producto y agregarlo al carrito
  3. Intentar "Place order" → error INSUFFICIENT FUNDS
  4. En Proxy → HTTP history se encuentran:
    • GET /api/products/1/price
    • POST /api/checkout
  5. Enviar GET /api/products/1/price al Repeater
  6. Cambiar el método GET → PATCH con un body JSON: 
    {"price": 0}
  7. La API acepta el PATCH y el precio del producto cambia a $0
  8. Agregar al carrito y completar la compra. Laboratorio completado.

Mass Assignment

¿Qué son los "Hidden Parameters"?

Son parámetros que la API acepta pero que no están documentados y normalmente no aparecen en la aplicación web normal.

Ejemplo típico: la app te permite cambiar solo username y email, pero en realidad la API también acepta isAdmin, role, price, discount, user_id, etc., aunque el frontend no los muestre.

Si se logra enviar esos parámetros ocultos, se puede cambiar el comportamiento de la aplicación.

Cómo encontrar parámetros ocultos

Herramienta Qué hace Cuándo usarla
Burp Intruder Prueba miles de nombres de parámetros automáticamente Cuando tenés tiempo y una buena wordlist
Param Miner Prueba hasta 65.000 nombres por request automáticamente Casi siempre (recomendado)
Content Discovery Busca contenido y parámetros no linkeados Para descubrir endpoints y params

Mass Assignment — la vulnerabilidad

Muchos frameworks (Laravel, Spring Boot, Django, Node.js, etc.) tienen una funcionalidad llamada "auto-binding" o "mass assignment": el framework automáticamente toma todos los parámetros que lleguen en el JSON y los copia al objeto interno.

Ejemplo real:

Un GET /api/users/123 devuelve:

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "isAdmin": false,
  "role": "user",
  "balance": 0
}

Un PATCH normal para actualizar el perfil envía:

{
  "username": "wiener",
  "email": "wiener@example.com"
}

Si la API es vulnerable a Mass Assignment, se pueden agregar los parámetros vistos en el GET:

{
  "username": "wiener",
  "email": "wiener@example.com",
  "isAdmin": true,
  "role": "admin",
  "balance": 999999
}

Si no hay validación adecuada, la aplicación los acepta y se obtiene acceso de administrador.

Cómo explotarlo paso a paso

  1. Buscá un endpoint que permita actualizar algo (PATCH o PUT)
  2. Realizá un GET del mismo recurso (ej: GET /api/users/me o /api/profile)
  3. Compará ambos JSONs — los campos que aparecen en el GET pero no se envían en el PATCH son candidatos a hidden parameters
  4. Agregá esos campos al request de actualización con valores convenientes
  5. Probá primero con valores inválidos ("isAdmin": "foo") para verificar que la aplicación los procesa
  6. Si confirma procesamiento, probá con el valor deseado ("isAdmin": true)

Diferencia importante: Hidden Parameter = parámetro que existe pero no se usa normalmente. Mass Assignment = vulnerabilidad que permite abusar de esos parámetros porque el framework los bindea automáticamente.

Resumen metodológico

  1. Crawl con Burp Scanner + Browser
  2. Buscar endpoints con /api/ y revisar todos los .js (usar JS Link Finder)
  3. Tomar endpoints interesantes → Repeater
  4. Probar todos los métodos HTTP (especialmente OPTIONS, POST, PUT, DELETE, PATCH)
  5. Probar cambiando Content-Type (JSON ↔ XML especialmente)
  6. Leer muy bien los errores y las respuestas — muchas veces dan pistas de otros parámetros o endpoints
