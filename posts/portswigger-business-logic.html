Business Logic Vulnerabilities — Fallas de Diseño y 2FA Broken Logic

Análisis de vulnerabilidades causadas por fallos en la lógica de negocio: exceso de confianza en controles del cliente, manipulación de precios en checkout y explotación de 2FA mal implementado mediante fuerza bruta. Laboratorios resueltos de PortSwigger Web Security Academy.

Business Logic 2FA Client-Side Controls PortSwigger

¿Qué son las vulnerabilidades de lógica de negocio?

Son fallos en el diseño o en la implementación de una aplicación que permiten a un atacante provocar un comportamiento que la aplicación no debería tener, pero que técnicamente funciona porque sigue las reglas del código tal como está escrito.

En otras palabras: la aplicación hace exactamente lo que el código le dice que haga, pero ese comportamiento es perjudicial para el negocio o para la seguridad.

Ejemplo sencillo: Imagina un sitio de e-commerce donde el flujo normal es:

  1. Agregar producto al carrito
  2. Ir al checkout
  3. Pagar
  4. Recibir el producto

Una vulnerabilidad de lógica de negocio podría permitirte saltarte el pago y recibir el producto igual, o cambiar el precio a $0, o usar un cupón ilimitadas veces aunque el sistema diga que es de un solo uso.

No es un error técnico clásico (como SQL Injection o XSS), sino un fallo en las reglas de negocio.

¿Por qué se llaman "de lógica de negocio"?

Porque la "lógica de negocio" son las reglas que definen cómo debe comportarse la aplicación en la vida real:

  • Un usuario normal no puede pedir un reembolso sin haber comprado.
  • No se puede reservar stock sin pagar.
  • No se puede modificar un pedido después de que fue aprobado.
  • Un cupón solo se puede usar una vez.

Cuando estas reglas no se implementan correctamente o no se validan en el servidor, surge la vulnerabilidad.

¿Cómo surgen estas vulnerabilidades?

Surgen principalmente por suposiciones incorrectas que hacen los desarrolladores sobre cómo van a usar la aplicación los usuarios:

  • "Los usuarios siempre van a seguir el flujo que diseñamos."
  • "Los usuarios solo van a interactuar a través del navegador/web normal."
  • "Los datos que llegan siempre van a ser válidos y con sentido."
  • "Nadie va a intentar estados raros o inusuales."

Estas suposiciones llevan a problemas como:

  1. Validaciones débiles o solo en el cliente (frontend) — El precio, la cantidad o el descuento se valida solo en JavaScript. Un atacante modifica la petición HTTP y envía lo que quiere.
  2. Falta de validación del flujo completo en el servidor — No se verifica que el usuario realmente pasó por todos los pasos obligatorios.
  3. No anticipar entradas inesperadas — Números negativos, cantidades muy grandes, strings raras, estados intermedios que no deberían existir, etc.
  4. Condiciones de carrera (race conditions) — Dos peticiones simultáneas que el servidor no maneja correctamente (por ejemplo, redimir el mismo cupón dos veces al mismo tiempo).
  5. Falta de modelado de amenazas — Nadie pensó: "¿Qué pasaría si un usuario malicioso hace X cosa que un usuario normal nunca haría?"
  6. Complejidad de la aplicación — Cuanto más grande y con más funcionalidades es la app, más fácil es que queden "agujeros" en las reglas.

¿Por qué son difíciles de detectar?

  • Los escáneres automáticos no las encuentran fácilmente, porque no hay un patrón técnico claro.
  • Solo se ven cuando alguien prueba manualmente pensando como un atacante.
  • Requieren conocimiento del dominio (entender cómo funciona el negocio).

Por eso son muy buscadas por bug bounty hunters y pentesters manuales.

Ejemplos comunes

  • Comprar un producto y cambiar el precio a 0 o negativo antes de pagar.
  • Redimir un cupón de "una sola vez" múltiples veces usando peticiones concurrentes.
  • Modificar el estado de un pedido después de que ya fue aprobado.
  • Acceder a datos o funciones de otro usuario cambiando un ID.
  • Reservar todo el stock sin pagar para que nadie más pueda comprar.

Excessive Trust in Client-Side Controls

Esta es una de las causas más comunes de las vulnerabilidades de lógica de negocio.

Los desarrolladores cometen un error grave: asumen que el usuario solo va a usar la aplicación tal como ellos la diseñaron (es decir, solo a través del navegador, haciendo clic en los botones normales y respetando las validaciones que se ven en pantalla).

Por eso confían demasiado en las validaciones que se hacen en el navegador (client-side), como:

  • JavaScript que controla el precio
  • Validaciones de formularios
  • Restricciones en los campos (máximo de cantidad, mínimo de caracteres, etc.)
  • Botones deshabilitados
  • Flujos guiados paso a paso

El problema fundamental es que todo lo que ocurre en el navegador el usuario lo puede controlar completamente.

Un atacante no está obligado a usar el sitio web como un usuario normal. Puede:

  • Interceptar las peticiones con herramientas como Burp Suite
  • Modificar cualquier dato después de que salió del navegador pero antes de que llegue al servidor
  • Enviar peticiones directamente con herramientas como Postman, curl o scripts en Python

Resultado: Todas las validaciones del lado del cliente se vuelven inútiles.

¿Cómo funciona en la práctica?

  1. El usuario llena un formulario en la página.
  2. El JavaScript valida que el precio sea $100, la cantidad máxima 5, etc.
  3. Al hacer clic en "Comprar", el navegador envía los datos al servidor.
  4. El atacante (con Burp Proxy) intercepta esa petición y cambia:
    • El precio de $100 → $1
    • La cantidad de 1 → 1000
    • El descuento de 10% → 999%
  5. El servidor recibe esos datos modificados y, si no valida correctamente, los acepta como válidos.

Lab: Excessive Trust in Client-Side Controls

Objetivo: Comprar un producto modificando el precio desde el cliente.

Al interceptar con Burp la petición al agregar el producto al carrito, se puede ver que el precio viene como parámetro en la petición. Como solo está validado del lado del cliente, se puede cambiar el precio a $1 y completar la compra. Laboratorio completado.

2FA Broken Logic

¿Qué vulnerabilidad tiene este lab?

Es un fallo grave de lógica en la implementación de Two-Factor Authentication (2FA).

La aplicación no verifica correctamente que la persona que está completando el paso de 2FA sea la misma que inició el login.

Específicamente:

  • Después de poner usuario + contraseña, la aplicación genera un código 2FA y lo envía por email.
  • En la petición que verifica el código (/login2), existe un parámetro llamado verify que indica a qué usuario pertenece el código que se está verificando.
  • La aplicación confía ciegamente en el valor de ese parámetro sin comprobar que coincida con la sesión iniciada.

Resultado: Podés iniciar login con tu cuenta (wiener), recibir tu código por email, y luego cambiar el parámetro verify a carlos para que el código se aplique a la cuenta de la víctima.

Esto es un clásico ejemplo de Broken Logic in Multi-Factor Authentication.

Resolución paso a paso

Paso 1 — Identificar el endpoint de 2FA:

Al iniciar sesión, se captura la petición a /login2. El parámetro verify contiene el nombre de usuario para el que se genera el código 2FA:

GET /login2
Cookie: verify=wiener; session=...

Se cambia verify=wiener a verify=carlos. La respuesta es 200 OK — esto confirma que se generó un código 2FA para el usuario carlos.

Paso 2 — Fuerza bruta del código 2FA:

La segunda petición a /login2 verifica el código:

POST /login2
Cookie: verify=carlos; session=...

mfa-code=XXXX

Se envía esta petición al Intruder (o Caido Automate) y se lanza un ataque de fuerza bruta en el campo mfa-code, iterando todos los valores de 0000 a 9999.

Paso 3 — Usar la sesión obtenida:

Una vez encontrado el código correcto, la respuesta incluye una Cookie de sesión válida para carlos. Se reemplaza la cookie en el navegador y se accede como carlos. Laboratorio completado.

Por qué esto es grave en entornos reales

  • El 2FA está diseñado para ser el segundo factor de seguridad. Si puede ser eludido cambiando un parámetro, la seguridad adicional es ilusoria.
  • En aplicaciones reales con rate limiting insuficiente, este ataque puede realizarse automáticamente.
  • Este tipo de fallo aparece en bug bounty con frecuencia y suele clasificarse como crítico (P1/Critical) ya que permite tomar control de cuentas ajenas.
